# ODT Format Fidelity Specification

## Reference Standard

**ODF 1.2** (OASIS Open Document Format for Office Applications, Version 1.2)
- Part 1: OpenDocument Schema
- Part 3: Packages

s1engine targets ODF 1.2 conformance as produced by LibreOffice and other major ODF implementations. The `office:version="1.2"` attribute is set on all generated content.

## Implementation Overview

The ODT format crate (`s1-format-odt`) provides read/write round-trip capability for the ODF text document format. The implementation uses `quick_xml` for XML parsing and `zip` for archive handling. Parsing is lenient (unknown elements skipped), while writing produces valid ODF 1.2 output.

### ZIP Structure

| Entry | Read | Write | Notes |
|-------|------|-------|-------|
| `mimetype` | Verified | Written first | Uncompressed, as required by ODF spec |
| `content.xml` | Full | Full | Document body + automatic styles |
| `styles.xml` | Full | Full | Named styles + page layout + master pages |
| `meta.xml` | Full | Full | Dublin Core metadata |
| `settings.xml` | Parsed | Written | View settings (zoom level) |
| `META-INF/manifest.xml` | Implicit | Full | Dynamic based on content |
| `Pictures/*` | Full | Full | Embedded images |

## ODF-Specific Architecture

### Style System

ODF has a two-tier style system that differs fundamentally from OOXML:

| ODF Concept | s1-model Mapping | Notes |
|-------------|------------------|-------|
| **Named styles** (`<style:style>` in `styles.xml`) | `Style` objects in `DocumentModel::styles()` | Paragraph, character, table styles |
| **Automatic styles** (`<style:style>` in `content.xml`) | Flattened into node `AttributeMap` | Per-node formatting, resolved at parse time |
| **Page layout** (`<style:page-layout>`) | `SectionProperties` | Page dimensions, margins, orientation |
| **Master pages** (`<style:master-page>`) | `SectionProperties` + Header/Footer nodes | Links page layout to header/footer content |

**Key design decision**: ODF automatic styles are resolved during parsing and merged directly into node attributes. They are NOT preserved as separate style objects. During writing, automatic styles are regenerated by collecting unique formatting combinations and emitting `<style:style>` elements with auto-generated names (e.g., `P1`, `T1`, `C1`).

### List Flattening

ODF represents lists as nested `<text:list>` / `<text:list-item>` structures. s1engine flattens these into paragraphs with `ListInfo` attributes:

```
ODF:                              s1-model:
<text:list>                       Paragraph { ListInfo { level: 0, ... } }
  <text:list-item>                Paragraph { ListInfo { level: 0, ... } }
    <text:p>Item 1</text:p>      Paragraph { ListInfo { level: 1, ... } }
    <text:list>
      <text:list-item>
        <text:p>Sub-item</text:p>
      </text:list-item>
    </text:list>
  </text:list-item>
</text:list>
```

During writing, the list nesting is reconstructed from the `ListInfo` level attributes using a depth-tracking stack.

### Unit Conversion

ODF uses CSS-like units (cm, mm, in, pt, px) while s1-model uses points internally. The `xml_util::parse_length()` function handles conversion:

| ODF Unit | Conversion |
|----------|------------|
| `cm` | 1cm = 28.3465pt |
| `mm` | 1mm = 2.83465pt |
| `in` | 1in = 72pt |
| `pt` | 1:1 |
| `px` | 1px = 0.75pt (96dpi assumption) |
| `%` | Handled contextually |

## ODF-Specific Known Limitations

1. **Single master page**: Only one master page is parsed. Documents with multiple master pages (e.g., landscape sections mid-document) will lose section-specific page layouts beyond the first.

2. **No mid-document section breaks**: ODF section breaks (`text:section`) are not parsed. The entire document uses a single page layout.

3. **Drawings dropped (ODT-11)**: All `draw:*` elements except `draw:frame` + `draw:image` are silently dropped. This includes custom shapes, text boxes, connectors, grouped objects, and SVG paths. This is a deliberate design decision documented as WONTFIX.

4. **Change tracking is raw XML only**: The `text:tracked-changes` block is captured as a raw XML string on the body node's `RawXml` attribute. Individual change regions are not semantically modeled. Inline change markers (`text:change-start`, `text:change-end`, `text:change`) are silently skipped.

5. **No embedded object support**: ODF embedded objects (math formulas, charts, spreadsheets) stored in subdirectories are not parsed. Only images in `Pictures/` are extracted.

6. **No column layout**: Multi-column layout (`style:columns`) in page layouts is not parsed.

7. **Automatic style names not preserved**: When writing, automatic styles are regenerated with new names (`P1`, `T1`, `C1`, etc.). The original ODF automatic style names are lost. This does not affect formatting fidelity but means XML-level byte-exact round-trips are not achievable.

8. **Internal hyperlink resolution**: Hyperlinks with `href="#bookmark-name"` are preserved as-is but not resolved to `BookmarkStart` node positions. Consumers must implement their own bookmark-href resolution (ODT-12).

9. **Limited font handling**: ODF font declarations (`<style:font-face>`) are mapped to font-family names but font properties (charset, family-generic, pitch) are not preserved.

10. **No conditional text**: ODF conditional text (`text:conditional-text`, `text:hidden-text`, `text:hidden-paragraph`) is not supported.

## Test Strategy

### Unit Tests (119 tests in `s1-format-odt`)

- **Reader tests**: Minimal ODT construction via in-memory ZIP, parse, verify model
- **Writer tests**: Build model programmatically, write to ODT, verify ZIP and XML
- **Content parser tests**: Paragraphs, headings, spans, lists, tables, images, annotations, footnotes, bookmarks, TOC, change tracking
- **Property parser tests**: Text properties, paragraph properties (borders, tab stops, line spacing)
- **Style parser tests**: Named styles, automatic styles, page layout, master pages, header/footer content
- **Content writer tests**: Body serialization, auto-style generation, list nesting reconstruction
- **Metadata parser/writer tests**: Dublin Core properties

### Round-Trip Tests

Round-trip testing validates: `ODT bytes -> read() -> DocumentModel -> write() -> ODT bytes -> read() -> DocumentModel`, then compare.

Current round-trip coverage:
- Basic text content
- Paragraph formatting (alignment, indentation, spacing, borders)
- Character formatting (bold, italic, underline, color, font, super/subscript)
- Styles (named, with inheritance, with attributes)
- Tables with formatting (borders, cell background, column widths, spans)
- Images (inline, with dimensions and alt text)
- Lists (bullet, numbered, multi-level, nested)
- Headers and footers (default, first page, with page number fields)
- Page layout (dimensions, margins, orientation)
- Metadata (title, creator, dates)
- Bookmarks (start, end, collapsed)
- Comments/annotations (with range markers)
- Footnotes and endnotes
- Table of contents
- Change tracking (raw XML preservation)

### Recommended Additional Tests

- Real-world ODT files from LibreOffice, Google Docs, and OpenOffice
- ODT files with complex automatic style chains
- Documents with multiple master pages (to verify graceful degradation)
- Documents with embedded objects (math, charts)
- Documents with drawings and shapes (to verify clean skipping)
- Large documents with many auto-styles
- Mixed-content scenarios (lists inside table cells, images in headers)
